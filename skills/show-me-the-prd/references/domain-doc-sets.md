# Domain Doc Sets — 도메인별 4종 문서 정의

> 4종 문서 개수는 사용자 예측가능성의 본질로 유지 (항상 4종 받음).
> 4종이 어떤 4종인지는 도메인에 맞게 가변.
> 사용자가 매번 "이번엔 몇 종 받지?" 고민할 필요 없음.

---

## 사용 방법

Phase 1 인터뷰에서 사용자 아이디어의 도메인을 추정한다:

1. 도메인이 일반/모호 → **기본 set** 사용
2. 도메인이 명확 (B2B SaaS / 1인 도구 등) → **도메인 특화 set** 사용
3. 도메인이 카탈로그에 없음 → 사용자에게 4종 자유 정의 인터뷰 (**개수 4는 유지**)

각 set은 README.md + 4종 문서 = 항상 5 파일 출력.

## 도메인이 카탈로그에 없는 경우

사용자에게 4종을 자유 정의받는다:

```
이 도메인은 카탈로그에 없네요. 4종 문서를 정의해주세요
(개수 4는 예측가능성의 본질이라 유지):

1. [요구사항 문서] — 무엇을 만들지
2. [구조 문서] — 데이터/모델/구조
3. [도메인 특화 문서] — 이 도메인 특수 사항 (규제/흐름/단계 등)
4. [AI 행동 규칙] — 개발 시 지켜야 할 규칙

사용자가 답변 못 하면 → 기본 set 사용
```

---

## ⚠️ 절대 규칙

1. **개수 4는 유지** — 사용자 예측가능성의 본질 (항상 4종 받음)
2. **README.md는 별도** — 4종 문서 + README 1 = 5 파일 출력 (네비게이션 보장)
3. **domain-doc-sets.md에 특정 회사/제품명 하드코딩 금지** — 도메인별 일반화만 (No-Site-Name Rule 차용)
4. **도메인 자동 추정이 모호하면 기본 set 사용** — 잘못된 도메인 추정보다 안전
5. **사용자가 명시적으로 도메인을 선택했을 때만** 도메인 set 적용 — Phase 1 인터뷰에서 확인
