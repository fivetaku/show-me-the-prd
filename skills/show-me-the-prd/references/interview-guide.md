# 인터뷰 가이드 -- 바이브코더에게서 정보 뽑는 법

## 바이브코더가 줄 수 있는 정보 vs 모르는 것

```
바이브코더가 아는 것:         바이브코더가 모르는 것:
├── 뭘 만들고 싶은지 (막연)    ├── 기술 스택
├── 누가 쓸 건지              ├── 데이터 구조
├── 어떤 게 불편했는지         ├── 인증/결제 방식
└── 대충 어떤 느낌이면 좋겠는지  ├── Phase 분리 기준
                             └── AI 제약 규칙
```

**AI가 해야 하는 일:**
1. **유저가 아는 것**을 질문으로 끌어낸다 (목적, 대상, 불만)
2. **유저가 모르는 것**은 AI가 조사해서 선택지로 제안한다 (기술, 구조, 방식)
3. 유저는 **설명을 읽고 선택만** 하면 된다

## AskUserQuestion 호출 규칙

### 필수 포함 요소

모든 옵션의 `description`에 반드시 포함:

| 요소 | 설명 | 예시 |
|------|------|------|
| 뭔지 | 이 선택이 뭘 의미하는지 | "구글/카카오 계정으로 로그인하는 방식이에요" |
| 장점 | 왜 이걸 고르면 좋은지 | "비밀번호 관리가 필요 없어서 편해요" |
| 단점 | 어떤 트레이드오프가 있는지 | "구글/카카오에 의존하게 돼요" |

### 옵션 배치 규칙

1. **(추천)** 옵션을 항상 **첫 번째**에 배치
2. 나머지 옵션은 **간단한 것 -> 복잡한 것** 순서
3. "잘 모르겠어요" 옵션은 AskUserQuestion이 자동 제공하는 "Other"로 대체 (별도 추가 불필요)
4. 최대 4개 옵션 (너무 많으면 선택 부담)

### 기술 용어 번역표

| 기술 용어 | 바이브코더 용어 |
|----------|---------------|
| OAuth / SSO | 소셜 로그인 (구글/카카오) |
| JWT / Session | 로그인 유지 방식 |
| Magic Link | 이메일로 로그인 링크 |
| WebSocket | 실시간 반영 (카카오톡처럼) |
| REST API | 서버와 데이터 주고받기 |
| ORM | 데이터 저장/조회 도구 |
| CI/CD | 코드 올리면 자동 배포 |
| SSL/TLS | 보안 연결 (자물쇠 표시) |
| CDN | 전 세계 어디서든 빠르게 접속 |
| Serverless | 서버 관리 없이 자동 확장 |
| Schema / Migration | 데이터 구조 설계도 |
| Middleware | 요청을 중간에서 검사하는 관문 |
| Container / Docker | 앱을 상자에 담아 어디서든 실행 |

### Markdown Preview 사용 규칙

AskUserQuestion의 `markdown` 필드를 활용해 시각적 프리뷰를 제공한다.

| 규칙 | 설명 |
|------|------|
| 사용 대상 | Step 3 (데이터 모델), Step 4 (Phase), Step 5 (기술 스택) -- 구조화된 정보를 보여줄 때만 |
| 비사용 대상 | Step 1 (아이디어), Step 2 (기능 선택) -- 단순 선호도/다중선택 질문에서는 불필요 |
| single-select 전용 | `multiSelect: true`에서는 markdown preview 미지원. Step 2 Q1 같은 다중선택에 사용 금지 |
| 콘텐츠 유형 | ASCII 관계도, 마크다운 테이블, 디렉토리 트리, 체크리스트 활용 |
| 동적 생성 | 하드코딩된 예시가 아닌, 실제 인터뷰 결과를 반영한 다이어그램을 매번 새로 생성 |
| 간결성 | 프리뷰는 핵심만. 15줄 이내 권장. 너무 길면 오히려 가독성 저하 |

**사용 예시:**

```
option:
  label: "좋아요, 이대로 갈게요 (추천)"
  description: "일반적인 구조라 확장하기 좋아요"
  markdown: |
    ```
    [User] ──1:N──> [Post] ──1:N──> [Comment]
    ```
    | 엔티티 | 핵심 필드 |
    |--------|----------|
    | User   | email, name |
    | Post   | title, content |
```

### 복잡도 표시

기능이나 선택지의 난이도를 직관적으로 전달:

| 표시 | 의미 | 언제 사용 |
|------|------|----------|
| 간단해요 | 기본 기능, 대부분 프레임워크에서 바로 됨 | CRUD, 파일 업로드, 폼 |
| 좀 복잡해요 | 외부 서비스 연동 또는 추가 설정 필요 | 결제, 알림, 검색 |
| 많이 복잡해요 | 전문 지식/비용 필요, 첫 버전에서 빼기 추천 | AI/ML, 실시간, 다국어 |

## Step별 정보 추출 전략

| Step | AI가 하는 일 | 유저에게 묻는 것 | 유저가 모르는 것은 |
|------|------------|----------------|-----------------|
| 1. 아이디어 | 경청 + 정리 | 뭘 만들어? 누가 써? 왜? | -- |
| 2. 기능 정리 | 유사 서비스 웹 리서치 -> 기능 목록 제시 | "이 중에 필요한 거?" | AI가 리서치해서 채움 |
| 2. 복잡도 | 각 기능의 기술 난이도 조사 | "이거 빼도 돼?" | AI가 판단해서 설명 |
| 3. 데이터 | 기능에서 자동 추출 | "이것도 저장해야 해?" | AI가 구조 제안 |
| 4. Phase | 복잡도 + 의존성 기반 자동 분리 | "이 순서 맞아?" | AI가 분리해서 확인만 |
| 5. 기술 스택 | 웹 리서치로 현재 트렌드 조사 | "이 중에 뭘로?" | AI가 조사해서 추천 |
| 6. 문서 생성 | 전체 종합 | -- | AI가 자동 생성 |
| 7. 안내 | 다음 단계 가이드 | "뭘 할까요?" | AI가 옵션 제시 |

## 도메인별 추가 질문 패턴

도메인을 인식하면 해당 분야에 특화된 질문을 추가한다.

| 도메인 | 추가 질문 |
|--------|----------|
| 이커머스 | 결제 수단? 배송 추적? 재고 관리? |
| 콘텐츠/미디어 | 파일 종류? 스토리지 용량? 공유 범위? |
| 생산성 | 협업? 알림? 외부 서비스 연동? |
| 소셜 | 팔로우/팔로잉? 피드 알고리즘? 신고/차단? |
| 교육 | 진도 추적? 퀴즈? 인증서? |
| 헬스케어 | 개인정보보호? 데이터 암호화? 의료법 규정? |
| 금융 | 결제 PG? 환불 정책? 세금? |

## Auto-memory 연동

**인터뷰 중 — 맥락 수집:**
- 선택 결과뿐 아니라 **왜 그걸 골랐는지** 이유를 파악한다
- "기존 앱의 뭐가 불만이었는지", "왜 이 기능이 필요한지" 등의 동기를 기억한다
- 이전 세션의 선호가 있으면 "(기억 기반 추천)" 라벨 추가

**완료 시 — 맥락 출력:**
- Step 7에서 단순 선택 목록이 아닌 **의도+이유 중심** 요약을 출력
- "로그인 제거 → 나만 쓸 거라 불필요" 같은 결정+근거 쌍이 핵심
- 이 맥락이 개발 단계에서 Claude의 마이크로 결정에 반영됨

## 대화 톤 가이드

- **친근하지만 전문적**: "~해요" 체 사용. 반말이나 과도한 존대 X.
- **확신 있게 추천**: "이걸 추천해요" (O) / "이게 좋을 수도 있어요" (X)
- **불확실할 때 솔직하게**: "이 부분은 확실하지 않아서 조사해볼게요"
- **과잉 설명 금지**: 한 옵션 설명은 2-3문장. 너무 길면 안 읽음.
- **숫자로 구체화**: "시간이 좀 걸려요" (X) / "2주 정도 더 걸려요" (O)
